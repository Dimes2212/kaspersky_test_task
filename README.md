# Kaspersky Test Task

## Что делает приложение

Приложение загружает пользователей с сервера в формате JSON и показывает их в двух представлениях:

- `Users` (`/users`) — таблица пользователей с поиском, сортировкой, добавлением и удалением.
- `Groups` (`/groups`) — те же пользователи, сгруппированные по группам, плюс отдельный блок пользователей без группы.

Также есть приветственная страница `Welcome` (`/`) с переходами на оба представления.

## Стек

- Frontend: `Next.js (App Router)`, `React`, `TypeScript`, `Tailwind CSS`
- Backend: `Express`, `TypeScript`
- Инструменты: `concurrently`, `tsx`

## Структура роутов

- `/` — приветственная страница
- `/users` — страница со списком пользователей (таблица)
- `/groups` — страница групп пользователей

## API

Express-сервер поднимается отдельно и работает с файлом `server/data/users.json`.

Доступные эндпоинты:

- `GET /api/health` — проверка сервера
- `GET /api/users` — получить всех пользователей
- `POST /api/users` — добавить пользователя
- `DELETE /api/users/:id` — удалить пользователя

## Как запустить

Требования:

- `Node.js 20+`
- `npm`

Установка зависимостей:

```bash
npm install
```

Запуск frontend + backend одновременно:

```bash
npm run dev
```

После запуска:

- Next.js: `http://localhost:3000`
- API: `http://localhost:3001`

Проверка API:

```bash
curl http://localhost:3001/api/health
```

## Бизнес-логика

### Страница Users (`/users`)

- отображение пользователей в виде таблицы с фиксированным набором колонок:
  `name`, `account`, `email`, `group`, `phone`, `age` (+ колонка действия удаления);
- загрузка данных при открытии страницы через `GET /api/users`;
- поиск работает по всем основным полям пользователя:
  `name`, `account`, `email`, `group`, `phone`, `age`;
- строка поиска нормализуется (обрезаются пробелы по краям, поиск без учета регистра);
- фильтрация выполняется на фронте и применяется к текущему списку пользователей;
- сортировка выполняется на фронте по выбранному режиму:
  - `name` (A-Я / Я-A);
  - `group` (A-Я / Я-A);
  - `age` (по возрастанию / по убыванию);
- сортировка применяется после фильтрации, чтобы пользователь видел упорядоченный результат поиска;
- добавление пользователя:
  - открывается модальное окно;
  - есть валидация полей (формат email, ограничения по возрасту, допустимые символы и обязательные поля);
  - при успешной проверке отправляется `POST /api/users`;
  - созданный пользователь добавляется в локальный список без полного перезапроса;
- удаление пользователя:
  - выполняется по кнопке в строке таблицы;
  - отправляется `DELETE /api/users/:id`;
  - при успешном ответе пользователь удаляется из локального списка;
- обработаны ошибки запросов:
  - при проблемах с `GET/POST/DELETE` на странице выводится сообщение об ошибке;
- модальное окно закрывается по кнопке отмены, успешному сохранению и клику вне окна.

Оптимизация:

- поиск и сортировка обернуты в `useMemo`;
- для поиска используется `useDeferredValue`;
- обработчики добавления/удаления мемоизированы через `useCallback`.

### Страница Groups (`/groups`)

- загружает тот же набор пользователей с API (`GET /api/users`);
- пользователи группируются на фронте по полю `group`;
- для каждой группы рисуется отдельный блок со списком пользователей;
- внутри группы пользователи сортируются по имени для стабильного отображения;
- пользователи с пустым `group` (или `null`) попадают в отдельный блок `Без группы`;
- если данных нет, выводится отдельное состояние `Пользователи не найдены`.

## Выводы по проектированию UI

- Страница `Users` разработана вручную без использования LLM.
- Страница `Groups` спроектирована с помощью LLM и затем адаптирована под проект.

Что касается разработки данного проекта при помощи LLM и нет, то они, конечно, разительно отличались. Количество затраченного времени при разработке мануально кратно превысило время разработки при помощи нейронки, но в то же время стоит понимать, что начало работы с нейросетью было уже после того как частично проект был уже готов и модели было уже проще ориентироваться в системе и не надо было придумывать что-то новое.
На мой взгляд, нейросети нужно отдавать лишь те задачи, которые во многом являются рутинными и не содержат в себе какой-либо значительной смысловой нагрузки. Основная разница во времени лишь заключалась в том, что компоненты не приходилось верстать самостоятельно, что отнимало много времени, начиная замыслом и заканчивая технической реализацией "на глаз".
